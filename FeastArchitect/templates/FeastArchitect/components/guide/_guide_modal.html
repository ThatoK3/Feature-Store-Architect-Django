The Four Core Components
🗄️

Data Sources

30+ supported databases including PostgreSQL, MongoDB, Kafka, Snowflake, and S3. Each with Debezium CDC support indicators and Spark integration patterns.

FileSource, KafkaSource, PushSource
👤

Entities

Business objects with unique join keys. The backbone of feature relationships enabling point-in-time correct joins across different feature views.

Entity(name="user", join_keys=["user_id"])
📊

Feature Views

Logical groupings of features with TTL, entities, and data sources. Support batch, streaming, and on-demand computation modes.

FeatureView, StreamFeatureView, OnDemandFeatureView
🚀

Feature Services

Model-ready feature bundles with versioning and SLA tracking. Composite services can reference other services for layered abstractions.

FeatureService(name="recommendation_v1")

Change Data Capture (CDC) Pattern

Best for: Transactional databases requiring near real-time features with historical accuracy.

  • Debezium captures row-level changes from PostgreSQL/MySQL transaction logs
  • Kafka provides durable event streaming with configurable retention
  • Spark Structured Streaming processes events into the offline store
  • Materialization syncs to online store for low-latency serving

Streaming-First Pattern

Best for: High-velocity event streams requiring sub-second feature freshness.

  • Direct Kafka-to-Feature-View ingestion using Spark Structured Streaming
  • Windowed aggregations with watermarks for late data handling
  • Dual write to online (low latency) and offline (historical) stores

Lambda Architecture Pattern

Best for: Complex features requiring both historical completeness and real-time updates.

  • Batch layer: Complete historical features with full accuracy
  • Speed layer: Real-time increments for immediate availability
  • Serving layer: Merged view combining both layers

Python SDK Usage Patterns
Batch Retrieval (Training)
from feast import FeatureStore
    import pandas as pd

    store = FeatureStore(repo_path=".")

    # Point-in-time correct join
    training_df = store.get_historical_features(
        entity_df=pd.DataFrame({
            "user_id": ["user_1", "user_2"],
            "event_timestamp": [
                datetime(2024, 1, 1),
                datetime(2024, 1, 2)
            ]
        }),
        features=[
            "user_demographics:age",
            "user_demographics:country",
            "transaction_stats:amount_30d"
        ]
    ).to_df()
Online Retrieval (Inference)
from feast import FeatureStore

    store = FeatureStore(repo_path=".")

    # Low-latency feature lookup
    features = store.get_online_features(
        feature_service="recommendation_v1",
        entity_rows=[
            {"user_id": "user_123", "context": "mobile"}
        ]
    ).to_dict()

    # On-demand transformation
    prediction = model.predict(
        features["user_demographics:age"],
        features["transaction_stats:amount_30d"]
    )
Feature Transformation Strategies
Pre-computed < 10ms

Transformations applied during ingestion or materialization. Best for: Aggregations, embeddings, complex SQL.

FeatureView with SQL transformation
On-Demand < 50ms

Computed at request time using request context. Best for: Time-since features, personalization logic.

OnDemandFeatureView with Python transform
Hybrid Variable

Combine pre-computed base features with on-demand overlays. Best for: Real-time personalization with historical context.

Feature Service composition
Performance Optimization
  • Online Store Selection

    Redis for sub-5ms latency, DynamoDB for AWS-native, Bigtable for Google Cloud. Consider data size vs. latency requirements.

  • Feature TTL Tuning

    Set aggressive TTLs for volatile features (1 hour for session data), longer for stable attributes (30 days for demographics).

  • Materialization Strategy

    Incremental materialization for large datasets. Schedule during off-peak hours for batch updates.

  • Embedding Caching

    Cache computed embeddings in the online store rather than computing on each request.

Conflict Detection & Resolution
    sequenceDiagram
        participant Client
        participant API
        participant DB
        
        Client->>Client: Compute MD5 hash of JSON
        Client->>API: PUT /repositories/123/ with client_hash
        API->>DB: Fetch current hash
        DB-->>API: server_hash
        
        alt Hash Match
            API->>DB: Update repository
            DB-->>API: Success
            API-->>Client: 200 OK
        else Hash Mismatch (Conflict)
            API-->>Client: 409 Conflict
            Note right of Client: Options:
1. Fetch latest (GET)
2. Force update (POST /force_update/)
3. Merge changes manually
        end

Hash-based optimistic locking prevents accidental overwrites. Each repository stores an MD5 hash of its architecture JSON. Updates include the client's hash; mismatches return 409 Conflict with resolution options.
